Harness mode — AGENTS.md parity

2026-07-23 · feature: harness-mode (proposed feat-005) · in-review · round 1

TL;DR

The cf-saas-starter-react-router AGENTS.md harness runs on three bespoke pieces brain-axi doesn't cover: a project-verification runner (init.sh + the typecheck/test/build steps of verify-done), task bookend runbooks (start-task / verify-done), and hand-written slash commands per repo. This plan adds brain verify (declared check registry + runner), two new playbooks (start, done), and generated .claude/commands/ wrappers — so a target repo's AGENTS.md becomes a thin pointer at brain commands and brain-axi is the whole harness, not just its memory.

Context

Per the 2026-07-14 checkpoint ("Turned .brain onto brain-axi itself… check 8/8, skill --check green"), brain-axi already dogfoods its own harness. feat-001 core-cli, feat-002 brain-review, and feat-003 shot-review are shipped; feat-004 annotation-watch is planned with a draft plan (2026-07-17-annotation-watch-runner). The latest cursor (2026-07-17) shows no feature in-progress — the one-in-progress slot is free.

The source harness is cf-saas-starter's AGENTS.md: a 5-subsystem harness (instructions / state / verification / scope / lifecycle) wired with init.sh, scripts/harness-check.sh, and four slash commands. Sean's ask: make brain-axi able to be that harness — same guarantees, brain commands instead of bespoke shell scripts.

One new primitive carries the whole design: the target repo declares its verification commands once in .brain/verify.json, and brain verify runs them deterministically — same TOON output, exit codes, and help: guidance as every other brain command. The bookend playbooks and generated slash commands are thin, generic wrappers around that primitive plus the commands that already exist.

Is this worth the weight? (round-1 concern, answered honestly)

Raised in review: "this might bloat brain-axi — or will it make it a better experience overall?" The three additions are not equal. Weighed one by one:

AdditionReal costReal valueVerdict
brain verify + registry ~150-200 lines in bin/brain.js (spawn, timeout, TOON table). No new deps, no new files in this repo beyond verify.json. Closes the verification subsystem — the one of the five harness concerns brain-axi currently can't touch. Evidence strings for ship/runs append come from a runner instead of agent transcription. Replaces a bespoke init.sh in every target repo. Also the CI story (brain verify && brain check). keep — core
start / done playbooks Pure text in the existing playbook table. Near-zero code weight; a sync burden only if command names change (skill --check already gates that). The bookends stop being hand-written per repo. This is exactly the contextual-disclosure pattern brain-axi already bets on. keep — cheap
setup --commands generation Highest bloat risk: four markdown templates baked into the single-file bin, and a second drift surface like skillContent() — every command rename now updates skill + playbooks + templates. Convenience only — a human-typeable entry point. Agents already get the same content from brain playbook start/done via the skill; a repo can hand-write 4-line wrappers in a minute. deferrable

Honest answer: verify is the product; command generation is the bloat. The revised recommendation (D6 below) ships Phases 1-2 now and defers Phase 3 until verify has proven itself in a real target repo — if playbooks turn out to be enough, Phase 3 dies and brain-axi stays lean. What keeps the CLI honest either way: zero-dep single file, TOON-only stdout, every addition gated by skill --check + brain check.

Decisions needed

0. Scope — how much of this ships?

OptionProsCons
All the harness power, none of the template-drift surface; smallest diff that makes brain-axi a full harness Target repos hand-write 4 thin slash-command files (minutes, once)
One-command install, full parity immediately Second generated-content drift surface in the bin
Zero new weight Verification subsystem stays outside the tool; every target repo keeps a bespoke init.sh + hand-remembered commands

1. Where does the check registry live? answered round 1: .brain/verify.json

The runner needs a machine-readable list of the project's real commands (typecheck, test, e2e, build, bootstrap).

OptionProsCons
Single-purpose, easy to validate, mirrors feature_list.json precedent; brain check gains a "parses" row One more file in the skeleton
No new file Muddies the feature tracker's single responsibility; write commands touch it constantly
Commands live next to their prose explanation Fragile parsing; humans editing prose can silently break the runner

2. Command surface for the runner?

init.sh has two modes (full bootstrap vs. --baseline) and verify-done has a third, heavier set (e2e, build). Checks are tagged with the stages they belong to.

OptionProsCons
One thing to learn; stages are data, not commands; default stage verify matches the common case Slightly longer invocations for baseline
Reads like init.sh Three COMMANDS entries + three help blocks for one feature; skill grows 3 sections
No new verb Conflates brain-integrity invariants with project tests; breaks "check is fast and read-only"

3. Should verify results be recorded into the brain?

The execute playbook's rule: evidence strings come from real command output. The runner is the natural producer of that evidence.

OptionProsCons
Evidence lands in deep state automatically when working a feature; harmless otherwise Agent must remember the flag (playbooks will say so)
Runner stays pure read/exec Re-introduces the transcription step the evidence rule exists to eliminate
Zero flags Surprising writes; wrong when verify runs outside feature work (CI, spot checks)

4. How do the four slash commands reach a target repo?

cf-saas-starter hand-writes .claude/commands/{start-task,verify-done,ship-feature,harness-check}.md. brain-axi can generate stack-agnostic versions that delegate to brain playbook start/done, brain verify, brain ship, brain check.

OptionProsCons
setup is already the "wire me into this repo" verb and already idempotent; one install step total setup grows a second concern
Clean separation Another top-level command for a once-per-repo action
No CLI change Non-deterministic output; drifts per repo; the exact problem brain skill --check solves for skills

5. Does the registry also own bootstrap (install / migrate), killing init.sh entirely?

OptionProsCons
init.sh fully replaced; one registry describes the project's whole command surface Mutating commands (install, migrate) run through brain — needs clear stage separation so verify stays side-effect-free
Runner stays strictly read-verify Parity incomplete; target repos still ship a bespoke script

The registry, concretely

Assuming D1/D2/D5 recommendations — .brain/verify.json in a target repo:

{
  "version": 1,
  "checks": [
    { "name": "install",   "run": "bun install",            "stages": ["bootstrap"] },
    { "name": "migrate",   "run": "bun run db:migrate:local","stages": ["bootstrap"] },
    { "name": "typecheck", "run": "bun run typecheck",      "stages": ["baseline", "verify"] },
    { "name": "unit",      "run": "bun run test",           "stages": ["baseline", "verify"] },
    { "name": "e2e",       "run": "bun run test:e2e",       "stages": ["verify"], "timeout": 600 },
    { "name": "build",     "run": "bun run build",          "stages": ["verify"] }
  ]
}

brain verify runs each matching check sequentially (they share caches and a DB — parallel is wrong here), captures exit code, duration, and the last ~15 lines of combined output, and emits a TOON table results[n]{check,status,exit,seconds} plus a tail: block per failure. Aggregate exit 1 on any failure; the help: list names the failing check and how to re-run just it (brain verify --only e2e). brain-axi's own verify.json registers node bin/brain.js skill --check and node bin/brain.js check --brain .brain — the CLI verifies itself with its own runner.

Plan of record

  1. Phase 1 — registry + runner. Add loadVerifyConfig + cmdVerify to bin/brain.js (--stage, --only, --feature per D3), plus a "verify.json parses (when present)" row in brain check. Acceptance: against brain-axi's own .brain, all stages run with correct exit codes; a bogus command yields a fail row + exit 1; missing/malformed registry produces guided opError.
  2. Phase 2 — bookend playbooks. Add playbook ids "start" (frame task → read brain → scope check → brain verify --stage baseline → run note → progress add) and "done" (brain verify → brain check → coherence sweep → close run note), generalized from cf-saas-starter's 00-before-task / 99-verify-done. Acceptance: brain playbook lists 5 ids; both render; content references only real commands.
  3. Phase 3 — generated slash commands. Per D4: setup writes stack-agnostic .claude/commands/{start-task,verify-done,ship-feature,harness-check}.md delegating to brain playbook/verify/ship/check, marker-guarded so hand-edited files are never clobbered. Acceptance: fresh scratch repo gets all four; re-run is a no-op; a user-modified file is skipped with a warning.
  4. Phase 4 — sync + dogfood + ship. Update skillContent() (new verify section + setup flag + playbook ids), add .brain/verify.json to this repo, update own .claude/commands to delegate to the new playbooks, feature MD + feature_list entry for feat-005. Acceptance: skill --check exit 0; brain check all-pass; brain ship harness-mode with runner-produced evidence.

FailureDetected byHandled byAgent sees

Malformed verify.json JSON.parse throw / schema shape check opError quoting the parse error; brain check row fails too exit 1 + the exact bad field
A check command exits non-zero child exit code fail row + last 15 output lines; remaining checks still run results table, aggregate exit 1, help: brain verify --only <name>
Check hangs past its timeout (default 300s, per-check override) timer child killed; status "timeout" timeout row, exit 1
