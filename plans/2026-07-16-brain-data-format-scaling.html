.brain data format & scaling strategy

2026-07-16 · feature: (harness-wide) · draft · round 0

TL;DR

The on-disk format is not the lever — output shaping is. Agents never read .brain/ files raw; they read the CLI's stdout, which is already TOON and already truncates long bodies, so switching storage to TOON or JSONC would save approximately zero context. The real scaling risk is the handful of commands whose output grows unbounded with data — above all brain context, which the SessionStart hook injects into every agent session and which currently prints every feature.

Context

Per the 2026-07-14 checkpoint ("Harness loop closed: .brain-onto-itself + .claude/commands verified. skill --check + brain check 8/8 green"), this brain now tracks brain-axi itself: feat-001 core-cli is shipped, feat-002 brain-review is in-progress. No prior plan artifacts exist — this is the first review round in this brain.

Today's data is small: 120 KB across 23 files, 2 features in feature_list.json (1.9 KB), 3 checkpoints in progress.md (1.9 KB). Nothing is slow yet. This plan is preventive: it decides the format question now and bounds the growth surfaces before a 50-feature, 200-checkpoint brain makes every session start expensive.

Decisions needed

1. On-disk storage format for .brain data?

The concern that prompted this plan: would TOON and/or JSONC storage scale better? Since agents only ever see CLI stdout, storage format changes zero context bytes — the tradeoffs are purely about tooling, diffs, and the zero-dependency constraint.

OptionProsCons
Native JSON.parse; git-diff friendly; humans and editors can open everything; zero migration No comments in the tracker (use description fields instead)
Inline comments in the tracker Node has no native JSONC parser — hand-rolled comment stripper under the zero-dep rule; other tools reading the file break; saves 0 context bytes
One format on the wire and at rest The codebase has a TOON encoder only — a parser must be written and maintained; worse diffs for nested data; no editor/ecosystem support; still saves 0 context bytes

2. What should home / context emit as the feature set grows? highest leverage

brain context runs from the installed SessionStart hook, so its output lands in every agent session unprompted. At 50 features, today's behavior injects a 50-row table before the agent has asked anything.

OptionProsCons
O(1) output regardless of brain size; the in-progress feature (the one that matters) stays fully visible; full list one command away Agent must run brain features to see everything
Minimal code change; still shows real rows Which N rows? Arbitrary cut hides shipped/blocked state the agent may need
No change Session-start context cost grows linearly with feature count, forever

3. How to handle progress.md growing forever?

Display is already capped (latest + 10), so the cost today is parse time and file size, not context. But a 500-checkpoint file makes every progress add rewrite a large file and every read re-split it.

OptionProsCons
Explicit, idempotent, CI-runnable next to brain check; archive stays searchable via brain search Someone has to run it (or wire it into a harness command)
Zero maintenance A write command silently moving data violates least-surprise; harder to reason about in reviews
Display already bounded; simplest File-size and parse cost grow forever; git churn on a huge file

4. Default limits for runs and docs lists?

OptionProsCons
Consistent with features/progress/search; the hint keeps truncation honest (AXI contextual disclosure) One more flag to keep in sync in skillContent()
Lists are one line per item — cheap in practice A brain with 300 run notes still dumps 300 lines

Plan of record

  1. Phase 1 — bound the session-start surfaces. Reshape cmdHome and cmdContext feature output per decision 2. Acceptance: against a generated 100-feature fixture, home and context output stays under ~30 lines; in-progress feature still shown in full; brain check and skill --check stay green.
  2. Phase 2 — default limits on remaining lists. Add --limit (default 20) + "N more" hint to runs and docs lists per decision 4; make search stop accumulating past its display limit. Acceptance: unknown flags still rejected (exit 2), every truncated output says so explicitly, --help updated per command.
  3. Phase 3 — progress compaction. Implement decision 3 (default: progress compact keeping newest 20, archiving to runs/archive/progress-YYYY.md). Acceptance: idempotent (second run is a no-op, exit 0); archived checkpoints still found by brain search; progress output unchanged after compaction.
  4. Phase 4 — sync the teaching surfaces. Update skillContent(), command help blocks, and CLAUDE.md for every changed command. Acceptance: brain skill --check exits 0; help: next-step lists mention the new flags where relevant.
