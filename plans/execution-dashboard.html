Execution dashboard — approval → PR

2026-07-16 · feature: brain-review · in-review · round 1 — D1–D4 locked (all recommended)

TL;DR

Once a plan is approved, the reviewer today loses visibility — execution progress lives in CLI TOON output and a sidebar strip coupled to the (now ended) review session. This plan adds a standalone live dashboard page on the existing review server that tracks a feature from plan approval through run steps, screenshots, and verification, ending at a visible "PR opened" terminal state. It reuses the server, SSE, file-watchers, and brain-data readers we already have — no new process, no new dependency.

Context

The 2026-07-14 checkpoint closed the harness self-setup loop ("skill --check + brain check 8/8 green"); feature feat-002 brain-review is the sole in-progress feature. The review architecture doc already anticipated this surface twice: Addendum v3 explicitly banked "a separate surface for browsing all plans and review sessions … possibly merged with the future execution-mode dashboard" as backlog, and Addendum v6.3 (decision D2) shipped a minimal execution strip inside the review session sidebar — feature status, health, last checkpoints, verification chips, shot thumbnails.

The gap: that strip dies with the session, shows summaries not run-step logs, and has no notion of "done" — nothing marks the moment a PR opens. This plan is the deferred v3 surface, scoped to a single feature's execution lifecycle.

plan approved→ in-progress→ run steps→ verification→ shipped→ PR opened

Big picture

Everything left of the server already exists (v6.4 execution verbs + feature-centric layout, v4). The new surface is the two /watch routes, one HTML page, one JS file, and a small PR-record write path.

All four decisions locked in round 1 (2026-07-16), each at the recommended option: D1 /watch/<feature> route on the existing server · D2 explicit brain pr verb · D3 step accordion · D4 end-of-session link + brain watch CLI. Cards kept below for the record.

Plan of record

  1. Phase 1 — data layer. brain-data.js: parse run-step files into structured steps ({n, title, observed}), read/write a per-feature PR record (features/<slug>/pr.json: {url, opened_at}), and a watchContext(brain, feature) payload = executionContext + steps + pr. Acceptance: node -e smoke calls return correct shapes against the fixture; brain check stays green.
  2. Phase 2 — server routes. server.js: GET /watch/<feature> (serves dashboard.html with {{FEATURE}} substituted), GET /watch/<feature>/context (watchContext JSON), GET /watch/<feature>/shot/<rel> (reuse existing path-sandboxed shot serving), SSE /watch-events/<feature> emitting context-update on .brain/features/<slug> changes (reuse existing watcher + debounce). Unknown feature → 404 JSON with help. Acceptance: curl returns payload; touching a run note fires SSE within 1s.
  3. Phase 3 — dashboard page. New lib/review/dashboard.html + dashboard.js (plain ES2020, editorial v7 palette): lifecycle pipeline header (stage chips, current stage bold), health strip, checkpoint feed, run-step accordion (per D3), verification verdict chips, screenshot rail, PR terminal card (link + opened_at) once pr.json exists. Empty states one-liners. Live-update on SSE. Acceptance: renders every section against the fixture brain-review feature; a runs append appears without manual reload.
  4. Phase 4 — CLI verbs + docs. bin/brain.js: `brain watch <feature>` (ensure server, print TOON {url}, open browser unless --no-open) and `brain pr <slug> --url <url>` (validates https URL, writes pr.json, appends progress checkpoint "PR opened for <slug>: <url>"). Both in COMMANDS with --help, unknown-flag rejection, help: next-steps. skillContent() + execute playbook updated (dashboard step + pr step after ship); skill --check green. Acceptance: verbs work end-to-end against fixture.
  5. Phase 5 — chrome handoff. chrome.js: when a session with a bound feature reaches status ended, render a "Watch execution →" link (to /watch/<feature>) in the ended-state banner. No other chrome changes. Acceptance: link appears only for feature-bound ended sessions.

Golden path

  1. Review session ends approved (feature-bound). Chrome shows "Watch execution →"; reviewer clicks it (or agent prints the URL from brain watch brain-review).
  2. Dashboard opens at /watch/brain-review: pipeline header shows plan approved stage; sections mostly empty-state.
  3. Agent runs the execute playbook: set-status in-progress → pipeline advances; each runs append lands as a new accordion step (verbatim output inside) within ~1s via SSE; each shots add adds a thumbnail; verification doc lands as a PASS/FAIL chip.
  4. brain ship brain-review --evidence "…" → stage shipped; health strip re-checks.
  5. Agent opens the PR (gh pr create), then brain pr brain-review --url https://github.com/... → dashboard renders the terminal PR opened card with the link. Loop complete: approval → visible execution → PR.
